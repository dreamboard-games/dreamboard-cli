---
title: "Public package surfaces"
description: "Use the canonical Dreamboard package imports and avoid internal paths."
---

Dreamboard intentionally keeps the author-facing package surface narrow. Prefer
generated workspace modules where they exist, and import generic SDK primitives
only from documented package subpaths.

## Reducer authoring

Import reducer helpers from the reducer subpath:

```ts
import {
  defineGame,
  defineGameContract,
  definePhase,
  defineInteraction,
  defineCardAction,
  defineEffect,
  defineView,
  defineStaticView,
  defineDerived,
  boardInput,
  cardInput,
  choiceTarget,
  formInput,
  promptInput,
  rngInput,
  pipe,
} from "@dreamboard/app-sdk/reducer";
```

`@dreamboard/app-sdk` does not expose a public root authoring surface. Do not
import from `@dreamboard/app-sdk/internal` or deep files under
`@dreamboard/app-sdk/reducer/*`.

## Manifest and workspace contracts

Use generated workspace contracts for game-specific ids and typed UI:

```ts
import { ids, literals, manifestContract } from "@dreamboard/manifest-contract";
import {
  WorkspaceGameShell,
  useInteractionByKey,
  useBoardInteractions,
  type InteractionParamsOf,
} from "@dreamboard/ui-contract";
```

Inside source paths that do not have package aliases, use the generated file
paths:

```ts
import { ids } from "../shared/manifest-contract";
import { WorkspaceGameShell } from "../shared/generated/ui-contract";
```

The rule is the same either way: authored code depends on generated workspace
contracts, not hand-written string ids or local copies of generated types.

## UI SDK

Use `@dreamboard/ui-sdk` for generic hooks, contexts, board helpers, and the
generic shell primitives:

```tsx
import {
  HexGrid,
  ResourceCounter,
  useGameView,
  useGameplayPhase,
  useIsMyTurn,
  usePluginSession,
} from "@dreamboard/ui-sdk";
```

Use `@dreamboard/ui-sdk/components` when you want the component-only subpath:

```tsx
import {
  BoardSurface,
  HandSurface,
  PanelSurface,
  DefaultInteractionButton,
} from "@dreamboard/ui-sdk/components";
```

New workspaces should not import board primitives from
`./components/dreamboard`. The scaffolded local module is for runtime wrappers
such as `PluginRuntime` and `ErrorBoundary` when a workspace keeps that local
re-export.

## Testing

Authored tests import from the generated wrapper:

```ts
import {
  defineBase,
  defineScenario,
  createTestRuntime,
} from "../testing-types";
```

The wrapper is generated by `dreamboard test generate` and narrows bases,
scenario interactions, views, phases, stages, and UI runtime helpers to the
current workspace.
