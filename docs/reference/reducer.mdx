`@dreamboard/app-sdk/reducer` is the public reducer authoring surface. It owns
game state schemas, phases, interactions, card actions, effects, views, static
views, derived values, setup, table reads, and table writes.

## Import surface

Import from the `./reducer` subpath:

```ts
import {
  defineGame,
  defineGameContract,
  definePhase,
  definePhaseStage,
  defineInteraction,
  defineCardAction,
  defineEffect,
  defineView,
  defineStaticView,
  defineDerived,
  createReducerBundle,
  createStateQueries,
  createTableQueries,
  boardInput,
  cardInput,
  choiceTarget,
  formInput,
  promptInput,
  rngInput,
  pipe,
  sparseCounts,
  sparseMap,
  type Op,
  type ReducerOps,
  type TableQueriesOfState,
} from "@dreamboard/app-sdk/reducer";
```

The package root is not an authoring surface. Do not import from
`@dreamboard/app-sdk`, `@dreamboard/app-sdk/internal`, or deep reducer files.

## Purity

Reducer callbacks must be deterministic. Do not call `Math.random()`,
`Date.now()`, network I/O, filesystem I/O, or process-global mutable state.

Use:

- `rngInput.*` when randomness belongs to a single player interaction submit
- `defineEffect` plus `fx.effect(...)` when runtime work should be queued and
  optionally resumed
- `ops.*` plus `pipe(...)` for table and state writes
- injected `q` for table reads

## Game contract

`defineGameContract(...)` binds the generated manifest contract to reducer
state and the legal phase-name tuple:

```ts
export const gameContract = defineGameContract({
  manifest: manifestContract,
  phaseNames: ["setup", "playerTurn"] as const,
  state: {
    public: z.object({
      winnerPlayerId: ids.playerId.nullable(),
    }),
    private: z.object({}),
    hidden: z.object({}),
  },
});
```

Use manifest-backed schemas from generated `ids` or `gameContract.schemas`.
For sparse maps keyed by manifest ids, use `z.partialRecord(...)`,
`sparseMap(...)`, or `sparseCounts(...)` instead of total enum records.

## Game assembly

`defineGame(...)` registers the contract, initial state, phases, views, optional
static view, derived values, and setup profiles:

```ts
export default defineGame({
  contract: gameContract,
  initialPhase: "setup",
  initialState: () => ({
    publicState: { winnerPlayerId: null },
    privateState: {},
    hiddenState: {},
  }),
  setupProfiles,
  phases,
  views: { player: playerView },
  staticView: boardStatic,
  derived: { scoreLeader },
});
```

`defineGame` checks that `contract.phaseNames`, `phases`, `initialPhase`, and
`setupProfiles[*].initialPhase` agree.

## Phases

`definePhase(...)` models one flow state:

```ts
export const playerTurn = definePhase<GameContract>()({
  kind: "player",
  state: playerTurnPhaseStateSchema,
  initialState: () => ({ step: "roll", diceRolled: false }),
  enter({ state, accept, ops, q }) {
    const active = state.flow.activePlayers[0] ?? q.player.order()[0]!;
    return accept(pipe(state, ops.setActivePlayers([active])));
  },
  actor: ({ state }) => state.flow.activePlayers,
  zones: [zones.actionHand],
  cardActions: { playActionCard },
  interactions: { rollDice, endTurn },
});
```

Use `kind: "player"` when the runtime waits for an interaction. Use
`kind: "auto"` for deterministic resolution phases. `enter(...)` runs when a
phase initializes or transitions in.

Stages are first-match gates inside a phase:

```ts
stages: {
  discard: definePhaseStage<GameContract, typeof phaseState>()({
    when: ({ state }) => state.phase.discardPending.length > 0,
    allow: ["discardCards"],
  }),
}
```

## Interactions

`defineInteraction(...)` is the canonical player-facing primitive:

```ts
export const tradeWithBank = defineInteraction<
  GameContract,
  typeof playerTurnPhaseStateSchema
>()({
  surface: "panel",
  label: "Trade with bank",
  group: "trade",
  step: "main",
  inputs: {
    giveResource: formInput.choice<ResourceId, GameState>({
      choices: "resourceMap",
    }),
    wantResource: formInput.choice<ResourceId, GameState>({
      choices: literals.resourceIds.map((id) => ({ value: id, label: id })),
    }),
  },
  validate({ input, q }) {
    if (!q.player.canAfford(input.playerId, { [input.params.giveResource]: 4 })) {
      return { errorCode: "INSUFFICIENT_RESOURCES" };
    }
    return null;
  },
  reduce({ state, input, accept, ops }) {
    return accept(
      pipe(
        state,
        ops.spendResources({
          playerId: input.playerId,
          amounts: { [input.params.giveResource]: 4 },
        }),
        ops.addResources({
          playerId: input.playerId,
          amounts: { [input.params.wantResource]: 1 },
        }),
      ),
    );
  },
});
```

The trusted bundle evaluates actor authorization, stage and step gates, costs,
availability, target validation, `validate`, and then `reduce`.

Important fields:

| Field | Purpose |
| --- | --- |
| `surface` | Routes descriptors to panel, board, hand, inbox, blocker, chrome, or custom UI. |
| `inputs` | Declares client params, target domains, defaults, and engine-sampled values. |
| `actor` | Overrides the phase actor for this interaction. |
| `to` | Addresses prompt/inbox interactions to one or more players. |
| `visibility` | `all` keeps non-actors visible but disabled; `actorsOnly` hides them. |
| `available` | Snapshot-time availability filter. |
| `validate` | Business-rule validation after params parse. |
| `cost` | Resource-cost metadata consumed by default renderers. |
| `reduce` | Returns `accept(...)` or `reject(...)`. |

Presentation metadata such as `label`, `shortLabel`, `icon`, `description`,
`emphasis`, `salience`, `group`, `presentation`, and `dispatchPriority` is
covered in [Interactions](/docs/authoring/interactions).

## Inputs and targets

Use input collectors instead of hand-written param parsing:

| Collector | Use |
| --- | --- |
| `formInput(schema)` | Free-form Zod-backed params. |
| `formInput.choice(...)` | One finite choice. |
| `formInput.choiceList(...)` | Multi-select finite choices. |
| `formInput.number(...)` | Numeric domains with min, max, and step. |
| `formInput.resourceMap(...)` | Resource counts with per-resource bounds. |
| `boardInput.vertex/edge/space/tile(...)` | Board target ids backed by a `boardTarget` rule. |
| `cardInput(...)` | Card id targets. |
| `promptInput(...)` | Addressed prompt response params, optionally backed by `choiceTarget`. |
| `rngInput.d6(...)` / `rngInput.coin()` | Engine-sampled values omitted from client params. |

Target rules drive both projection and submit validation. Do not duplicate
target legality in React.

## Card actions

Use `defineCardAction(...)` when the selected card is the object being played:

```ts
export const playActionCard = defineCardAction<
  GameContract,
  typeof playerTurnPhaseStateSchema
>()({
  label: "Play action card",
  playFrom: zones.actionHand,
  cardType: cardTypes.boost,
  inputs: {
    spaceId: boardInput.space<GameState, SpaceId>({
      target: selectableSpaceTarget,
    }),
  },
  validate({ state }) {
    return state.phase.actionCardPlayed
      ? { errorCode: "ACTION_CARD_ALREADY_PLAYED" }
      : null;
  },
  reduce({ state, input, accept, ops }) {
    return accept(
      pipe(
        state,
        ops.moveComponentToSpace({
          componentId: "shared-token",
          boardId: "main-board",
          spaceId: input.params.spaceId,
        }),
        ops.patchPhaseState({ actionCardPlayed: true }),
      ),
    );
  },
});
```

Register card actions on the phase and list the manifest player zone in
`zones`. The UI hand surface then receives playable descriptors for cards in
that zone.

## Effects

Effects are runtime instructions for engine work such as rolling a die or
shuffling a shared zone:

```ts
const rollProduction = defineEffect<GameContract>()({
  type: "rollDie",
  id: "roll-production",
  context: z.object({ reason: z.string() }),
  reduce({ state, input, accept, ops }) {
    return accept(
      pipe(
        state,
        ops.patchPublicState({
          lastRoll: input.response.value,
          lastRollReason: input.data.reason,
        }),
      ),
    );
  },
});

return accept(state, [
  fx.effect(rollProduction, {
    dieId: "production-die",
    context: { reason: "turn-start" },
  }),
]);
```

Effects are not prompts. Use `promptInput` plus `to` for player responses.

## Views and static views

Use `defineView(...)` for dynamic player-facing payloads:

```ts
export const playerView = defineView<GameContract>()({
  project({ state, playerId, q, derived }) {
    return {
      myResources: q.player.resources(playerId),
      activePlayerIds: state.flow.activePlayers,
      winnerPlayerId: state.publicState.winnerPlayerId,
      leader: derived(scoreLeader),
    };
  },
});
```

Use `defineStaticView(...)` for immutable session-scoped data such as board
topology:

```ts
export const boardStatic = defineStaticView<GameContract>()({
  project({ q }) {
    return {
      board: q.board.hex("main-board"),
    };
  },
});
```

Dynamic views can read state and `playerId`; static views cannot. Keep large
manifest-derived board topology out of per-action dynamic view updates.

## Derived values

Use `defineDerived(...)` for memoized pure computations shared by reducer
callbacks and views:

```ts
export const scoreLeader = defineDerived<GameContract>()({
  compute({ state, q }) {
    return q.player.order().find((playerId) => {
      return state.publicState.progressByPlayer[playerId] >= 10;
    }) ?? null;
  },
});
```

Read derived values through the injected `derived` resolver.

## Reads and writes

Reducer callbacks receive:

- `q` for typed table reads
- `ops` for curried state/table writes
- `accept` and `reject` for reducer results
- `fx` for runtime instructions
- `derived` for derived values

```ts
const hand = q.zone.playerCards(input.playerId, "action-hand");
const board = q.board.hex("main-board");
const vertices = q.board.spaceVertices("main-board", input.params.spaceId);

return accept(
  pipe(
    state,
    ops.spendResources({ playerId: input.playerId, amounts: COST }),
    ops.moveComponentToVertex({
      componentId: "marker",
      boardId: "main-board",
      vertexId: input.params.vertexId,
    }),
    ops.patchPhaseState({ step: "main" }),
  ),
);
```

Use `createStateQueries(...)`, `createTableQueries(...)`, or
`createReducerOps(...)` only when you are outside a reducer callback and do not
already have injected helpers.

## Bundle

`createReducerBundle(game)` is the runtime boundary used by the host, CLI test
harness, and generated testing runtime. Authored workspaces normally do not
call bundle methods directly except in generated or framework code.
