This tutorial walks through a deliberately small Dreamboard game so you can see
the whole authoring loop once: rules, manifest, reducer, UI, and tests.

The example game is `Race to Ten`.

- 2 players
- one shared d6
- each turn, the active player rolls the die and adds the result to their score
- the first player to reach 10 points wins

The reducer itself stays pure. Dice rolls run through `fx.rollDie`, a
runtime-owned effect that uses the engine's seeded RNG. Because every test base
pins a `seed`, scenarios are reproducible without any custom randomness
plumbing in your reducer.

## What you will build

By the end of the tutorial you will have:

- a `rule.md` that explains the game
- a `manifest.ts` with a shared die
- reducer code in `app/`
- a playable `ui/App.tsx`
- a base and scenario under `test/`

## Prerequisites

- Dreamboard CLI installed: `npm install -g dreamboard`
- authenticated with `dreamboard login`

## 1. Create the workspace

```bash
dreamboard new race-to-ten --description "A tiny scoring game with one shared die"
cd race-to-ten
```

The scaffold gives you the files you will edit next:

- `rule.md`
- `manifest.ts`
- `app/game-contract.ts`
- `app/game.ts`
- `app/phases/*`
- `ui/App.tsx`
- `test/bases/*`
- `test/scenarios/*`

## 2. Write `rule.md`

```md
# Race to Ten

## Overview

- Players: 2
- Objective: be the first player to reach 10 points
- Duration: 3 to 5 minutes

## Components

- 1 shared six-sided die
- visible score totals for each player
- no hidden information

## Setup

1. Seat two players.
2. Set both scores to 0.
3. Clear the die value.
4. Player 1 takes the first turn.

## Gameplay

### Phase 1: takeTurn

- Acting player: the current player only
- Allowed actions: `rollDie`
- Validation: only the active player may act, and no actions are legal after a winner exists
- Completion:
  - `rollDie` sets the shared die to a new value
  - the acting player adds that value to their score
  - if the acting player reaches 10 points, the game ends immediately
  - otherwise the turn passes to the other player

## Scoring and progression

- `rollDie` increases the acting player's score by the rolled value

## Winning conditions

- End trigger: a player reaches 10 points
- Winner determination: the player who reached 10 points wins
- Tie-breaker: not applicable because turns resolve one at a time

## Special rules and edge cases

- Actions after game end are illegal
- Out-of-turn actions are illegal
```

For a fuller reference, see [Rule authoring](/docs/reference/rule-authoring).

## 3. Write `manifest.ts`

This game needs player-count metadata and one shared die.

```ts
import { defineTopologyManifest } from "@dreamboard/sdk-types";

export default defineTopologyManifest({
  players: {
    minPlayers: 2,
    maxPlayers: 2,
    optimalPlayers: 2,
  },
  dieTypes: [
    {
      id: "standard-d6",
      name: "Standard d6",
      sides: 6,
    },
  ],
  dieSeeds: [
    {
      id: "turn-die",
      name: "Turn die",
      typeId: "standard-d6",
    },
  ],
});
```

Run:

```bash
dreamboard sync
dreamboard compile
```

`dreamboard sync` refreshes generated contracts from authored files.
It also prepares workspace dependencies after `package.json` changes.
`dreamboard compile` builds the current authored head.

## 4. Define the reducer contract

Open `app/game-contract.ts` and replace the empty schemas with the state the
rules require.

```ts
import { z } from "zod";
import * as manifestContract from "../shared/manifest-contract";
import {
  defineGameContract,
  type GameStateOf,
} from "@dreamboard/app-sdk/reducer";

const playerId = manifestContract.ids.playerId;

const publicStateSchema = z.object({
  winnerPlayerId: playerId.nullable(),
  lastRoll: z.number().int().min(1).max(6).nullable(),
  scores: z.record(playerId, z.number().int().nonnegative()),
});

const privateStateSchema = z.object({});

const hiddenStateSchema = z.object({});

export const gameContract = defineGameContract({
  manifest: manifestContract.manifestContract,
  // Keep this list in sync with the keys of the `phases` record in ./phases.
  // Declaring phase names here narrows `fx.transition(...)`, `phase.dispatch`,
  // and the flow state's `currentPhase` to a literal union.
  phaseNames: ["takeTurn"] as const,
  state: {
    public: publicStateSchema,
    private: privateStateSchema,
    hidden: hiddenStateSchema,
  },
});

export type GameContract = typeof gameContract;
export type GameState = GameStateOf<GameContract>;
```

The public state holds player-visible game progress. Race to Ten has no hidden
information, so the hidden state is an empty object — real games use it for
deck orders, secret goals, or other reducer-private fields.

Note what is not in `publicState`: whose turn it is, or the current die face.
Turn routing is owned by the engine as `state.flow.activePlayers` and is
updated through `ops.setActivePlayers(...)`. The rolled die value lives in
`state.table.dice["turn-die"].value`, which the runtime maintains when
`fx.rollDie(...)` fires. Keeping both out of `publicState` avoids two sources
of truth.

## 5. Add a player view

Create `app/player-view.ts`:

```ts
import { defineView } from "@dreamboard/app-sdk/reducer";
import type { GameContract } from "./game-contract";

export const playerView = defineView<GameContract>()({
  project({ state, playerId }) {
    return {
      activePlayerId: state.flow.activePlayers[0] ?? null,
      winnerPlayerId: state.publicState.winnerPlayerId,
      lastRoll: state.publicState.lastRoll,
      scores: state.publicState.scores,
      targetScore: 10,
      turnDieId: "turn-die",
    };
  },
});
```

`activePlayerId` is projected from `state.flow.activePlayers`, which the
engine owns. The view is free to expose whatever shape the UI needs — it is
not limited to `publicState` fields.

The projected payload type is inferred from `project`'s return value — no Zod
schema is required. Reducer callbacks inside `project(...)` also receive `q`
for reducer-native reads such as `q.zone.playerCardCollection(...)` or
`q.board.tiled(...)` when the view grows.

The UI will read this view instead of reconstructing game logic in React.

## 6. Implement the phase and action

Create `app/phases/take-turn.ts`:

```ts
import { z } from "zod";
import {
  defineAction,
  defineFlow,
  definePhase,
  pipe,
} from "@dreamboard/app-sdk/reducer";
import type { GameContract } from "../game-contract";

const TARGET_SCORE = 10;
const TURN_DIE_ID = "turn-die" as const;

const onDieRolled = defineFlow<GameContract>()({
  id: "on-die-rolled",
  type: "rollDie",
  reduce({ state, input, accept, ops, q }) {
    const actingPlayerId = state.flow.activePlayers[0];
    if (actingPlayerId == null) {
      return accept(state);
    }

    const rollValue = input.response.value;
    const nextScore =
      state.publicState.scores[actingPlayerId] + rollValue;
    const reachedTarget = nextScore >= TARGET_SCORE;

    const order = q.player.order();
    const nextPlayerId =
      order[(order.indexOf(actingPlayerId) + 1) % order.length]!;

    return accept(
      pipe(
        state,
        ops.patchPublicState((prev) => ({
          ...prev,
          winnerPlayerId: reachedTarget ? actingPlayerId : null,
          lastRoll: rollValue,
          scores: { ...prev.scores, [actingPlayerId]: nextScore },
        })),
        ops.setActivePlayers(reachedTarget ? [] : [nextPlayerId]),
      ),
    );
  },
});

const rollDie = defineAction<GameContract>()({
  params: z.object({}),
  validate({ state, input }) {
    if (state.publicState.winnerPlayerId) {
      return {
        errorCode: "GAME_ALREADY_ENDED",
        message: "The game has already ended.",
      };
    }

    if (!state.flow.activePlayers.includes(input.playerId)) {
      return {
        errorCode: "NOT_YOUR_TURN",
        message: "It is not your turn.",
      };
    }

    return null;
  },
  reduce({ state, accept, fx }) {
    return accept(state, [
      fx.rollDie(TURN_DIE_ID, { resume: onDieRolled() }),
    ]);
  },
});

export const takeTurn = definePhase<GameContract>()({
  kind: "player",
  state: z.object({}),
  initialState: () => ({}),
  enter({ state, accept, ops, q }) {
    if (state.publicState.winnerPlayerId) {
      return accept(state);
    }

    const firstPlayerId = q.player.order()[0]!;
    return accept(pipe(state, ops.setActivePlayers([firstPlayerId])));
  },
  actions: {
    rollDie,
  },
  flows: {
    onDieRolled,
  },
});
```

Two ideas drive this phase.

**Randomness belongs to the runtime.** The reducer never calls `Math.random()`
or writes into `state.table` directly. `rollDie.reduce(...)` queues one typed
effect: `fx.rollDie("turn-die", { resume: onDieRolled() })`. The `resume`
token is required — the SDK will not let you call `fx.rollDie(...)` without a
flow to handle the result. The engine rolls the die with its seeded
RNG, writes the new face into `state.table.dice["turn-die"].value`, and then
calls `onDieRolled.reduce(...)` with the rolled value surfaced as
`input.response.value`. Scoring logic reacts to that value instead of
computing one by hand.

**Turn routing belongs to the engine.** `ops.setActivePlayers(...)` is the
single source of truth for who can act next, and the engine exposes it back to
reducer callbacks as `state.flow.activePlayers`. `validate(...)` reads it to
reject out-of-turn actions, `enter(...)` seeds the first active player from
`q.player.order()`, and `onDieRolled.reduce(...)` passes the turn to the next
seat — or hands control to `[]` when a winner exists so the engine stops
routing actions. Nothing is mirrored into `publicState`.

`ops`, `q`, and `fx` are injected into every reducer callback. Destructure
them alongside `state`, `input`, and `accept` as you need them. Import `pipe`
directly from `@dreamboard/app-sdk/reducer` to compose `ops.*` writes.

App-owned helpers that need `ops` or `q` belong in `app/reducer-support.ts`
and should accept the injected values as arguments. The scaffold exports
`type Ops = ReducerOps<GameState>` and `type Q = TableQueriesOfState<GameState>`
so helpers can share a single source of truth:

```ts
import type {
  Op,
  ReducerOps,
  TableQueriesOfState,
} from "@dreamboard/app-sdk/reducer";
import type { GameState } from "./game-contract";

export type Ops = ReducerOps<GameState>;
export type Q = TableQueriesOfState<GameState>;
```

Then update `app/phases/index.ts`:

```ts
import { takeTurn } from "./take-turn";

export const phases = {
  takeTurn,
};
```

## 7. Register the game

Update `app/game.ts`:

```ts
import { defineGame } from "@dreamboard/app-sdk/reducer";
import { records } from "../shared/manifest-contract";
import { gameContract } from "./game-contract";
import { phases } from "./phases";
import { playerView } from "./player-view";

export default defineGame({
  contract: gameContract,
  initial: {
    public: () => ({
      winnerPlayerId: null,
      lastRoll: null,
      scores: records.playerIds(0),
    }),
    private: () => ({}),
    hidden: () => ({}),
  },
  initialPhase: "takeTurn",
  phases,
  views: {
    player: playerView,
  },
});
```

At this point the rules exist, but the UI is still the scaffold placeholder.

## 8. Build the UI

Update `ui/App.tsx`:

```tsx
import {
  useActions,
  useGameView,
  useIsMyTurn,
} from "@dreamboard/ui-sdk";
import { DiceRoller } from "./components/dreamboard";

export default function App() {
  const view = useGameView();
  const isMyTurn = useIsMyTurn();
  const phase = useActions();
  const diceValues = view.lastRoll == null ? [] : [view.lastRoll];

  const rollDie = async () => {
    if (phase.phase !== "takeTurn") {
      return;
    }

    await phase.dispatch(phase.commands.rollDie());
  };

  return (
    <main style={{ padding: 24, fontFamily: "system-ui, sans-serif" }}>
      <h1>Race to Ten</h1>
      <p>First to {view.targetScore} points wins.</p>

      <DiceRoller
        values={diceValues}
        diceCount={1}
        render={({ values }) => (
          <div style={{ fontSize: 32, marginBottom: 16 }}>
            Die: {values?.[0] ?? "?"}
          </div>
        )}
      />

      <ul>
        {Object.entries(view.scores).map(([playerId, score]) => (
          <li key={playerId}>
            {playerId}: {score}
            {view.activePlayerId === playerId ? " <- current player" : ""}
          </li>
        ))}
      </ul>

      <p>Last roll: {view.lastRoll ?? "not rolled yet"}</p>

      {view.winnerPlayerId ? (
        <p>Winner: {view.winnerPlayerId}</p>
      ) : (
        <button onClick={() => void rollDie()} disabled={!isMyTurn}>
          Roll die
        </button>
      )}
    </main>
  );
}
```

The important part is that:

- the die exists in authored game structure
- `fx.rollDie(...)` is the only code path that writes the die value; the
  reducer never touches `state.table` directly
- scoring happens in `onDieRolled.reduce(...)` once the runtime has rolled the
  die, so it can read `input.response.value` and update the public state in a
  single `pipe`
- the UI reads projected view data and the runtime die face through
  `view.lastRoll`, and dispatches actions through `useActions()`

For authored reducer reads, prefer the injected `q` on reducer callbacks
(`reduce({ state, q, ... })`, `project({ state, q, ... })`) and use
`q.board.*`, `q.zone.*`, `q.card.*`, or `q.component.*`. Shared helpers that
need `ops` or `q` belong in `app/reducer-support.ts` and should accept the
injected values as arguments.

## 9. Add a base and a scenario

Replace `test/bases/initial-turn.base.ts`:

```ts
import { defineBase } from "../testing-types";

export default defineBase({
  id: "initial-turn",
  seed: 1337,
  players: 2,
  setup: async ({ game }) => {
    await game.start();
  },
});
```

Create `test/scenarios/first-roll-scores.scenario.ts`:

```ts
import { defineScenario } from "../testing-types";

export default defineScenario({
  id: "first-roll-scores",
  description:
    "Player 1's opening roll is recorded, scored, and passes the turn to player 2.",
  from: "initial-turn",
  when: async ({ game }) => {
    await game.action("player-1", "rollDie");
  },
  then: ({ state, view, expect }) => {
    expect(state()).toBe("takeTurn");

    const p1 = view("player-1");
    const roll = p1.lastRoll ?? 0;
    expect(roll).toBeGreaterThanOrEqual(1);
    expect(p1.scores["player-1"]).toBe(roll);
    expect(p1.scores["player-2"]).toBe(0);
    expect(p1.winnerPlayerId).toBeNull();
    expect(p1.activePlayerId).toBe("player-2");
  },
});
```

The assertions intentionally avoid hard-coding the die face. The base pins
`seed: 1337`, so the roll is reproducible across runs, but the test stays
robust if you later change the seed: as long as the engine rolled a `1..6`
value, the acting player's score must equal `lastRoll`, player 2 must not
have scored, and the turn must have passed.

Generate artifacts and run the test suite:

```bash
dreamboard test generate
dreamboard test run
```

## 10. Run the game locally

Start the local dev server to play the game in the browser:

```bash
dreamboard dev
```

If you edit `rule.md` or `manifest.ts`, run `dreamboard sync` again before
continuing.

## Where to go next

This tutorial uses the smallest die-based loop that still touches authored
components, reducer state, UI, and tests. The next layer of depth is:

- add setup profiles or setup options
- add richer reducer state and more than one phase
- replace the plain button UI with grouped action panels and richer views
- add rejection-path tests such as out-of-turn actions

The tutorial already uses one runtime-owned effect (`fx.rollDie(...)`). When
you reach for more randomness, keep it on the same side of the line:

- `fx.rollDie(dieId, { resume })` to roll an authored die with seeded RNG and
  react to the rolled value through a typed flow
- `fx.shuffleSharedZone(zoneId, { resume })` to randomise a deck, then react
  to the new card order — or deal from the top with
  `ops.dealCardsToPlayerZone(...)` in the follow-up reducer
- `fx.open(promptFlow, { to, ... })` for a player choice that drives a
  reducer reaction once the player responds

Do not call `Math.random()` or `Date.now()` inside reducer callbacks — see
[Reducer](/docs/reference/reducer) for the full list of purity rules.
