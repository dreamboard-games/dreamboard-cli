# Test Harness

TypeScript scenario tests run against the Dreamboard backend API using deterministic base snapshots.

Use this file for the TypeScript regression harness:

- `dreamboard test generate`
- `dreamboard test run`

This is not the same system as `dreamboard run --scenario <file>.json`.

- For JSON runtime scenarios, see [scenario-format.md](scenario-format.md).
- For rejection-focused runtime coverage, see [adversarial-testing.md](adversarial-testing.md).

## Workspace Layout

The harness expects this structure in the game project:

- `test/base-scenarios.json`
- `test/scenarios/*.scenario.ts`
- `test/generated/*` (generated by `dreamboard test generate`)

Current canonical example workspace (`examples/things-in-rings`) includes bases like `initial-turn` and `after-first-placement`, plus scenarios under `test/scenarios/`.

## Commands

Generate deterministic bases:

```bash
dreamboard test generate
```

Run all scenarios:

```bash
dreamboard test run
```

Run one scenario file:

```bash
dreamboard test run --scenario test/scenarios/reject-card-not-in-hand.scenario.ts
```

Source-checkout builds may expose internal environment overrides for harness work, but those flags are not part of the published CLI contract.

Regenerate whenever any fingerprint input changes: base definition (`seed`, `players`, `steps`), game manifest hash, compiled result ID, or game ID.

## Base Definitions

`test/base-scenarios.json` is required and must explicitly include `initial-turn`.

Each base entry has:

- `seed: number`
- `players: number`
- `steps: ApiScenarioStep[]`

`steps` may be empty. Use an empty-step base plus an empty-step scenario to assert opening-state invariants without inventing fake actions.

Example:

```json
{
  "initial-turn": {
    "seed": 1337,
    "players": 4,
    "steps": []
  },
  "after-first-placement": {
    "seed": 1337,
    "players": 4,
    "steps": [
      {
        "playerId": "player-2",
        "actionType": "placeThing",
        "parameters": { "cardId": "a-shadow", "ringId": "ring-1" }
      }
    ]
  }
}
```

Generation flow for each used base:

1. Create session with deterministic seed/player count.
2. Start game.
3. Wait for first `YOUR_TURN`.
4. Execute base setup steps.
5. Save snapshot + fingerprint metadata.

Base setup events are not counted in test assertions. `events` in `then.assert` only reflects actions from `when.steps`, so zero-step scenarios start with an empty event history.

## Generated Artifacts

`dreamboard test generate` writes:

- `test/generated/base-states.generated.ts`
- `test/generated/base-states.generated.d.ts`
- `test/generated/scenario-manifest.generated.ts`
- `test/generated/.generation-meta.json`

These files are generated and should not be edited manually.

## Scenario Contract

Scenarios are loaded from `test/scenarios/**/*.scenario.ts` and validated strictly (unknown fields are rejected).

In current projects, scenario files typically import from `../testing-types`:

```ts
import { AssertContext, defineScenario } from "../testing-types";

export default defineScenario({
  meta: {
    id: "reject-card-not-in-hand",
    description: "Player tries to place a card they don't hold",
  },
  given: {
    base: "initial-turn",
  },
  when: {
    steps: [
      {
        playerId: "player-2",
        actionType: "placeThing",
        parameters: { cardId: "a-dog", ringId: "ring-1" },
      },
    ],
  },
  then: {
    assert: ({ gameState, events, expect }: AssertContext) => {
      expect(events.count("ACTION_REJECTED")).toBe(1);
      expect(events.last("ACTION_REJECTED")?.errorCode).toBe(
        "CARD_NOT_IN_HAND",
      );
      expect(gameState.currentState).toBe("placeThing");
    },
  },
});
```

`given.base` must exist in `test/base-scenarios.json`, or the run fails with an available-base list.

For invariant coverage, add at least one zero-step scenario:

```ts
import {
  AssertContext,
  defineScenario,
  getNormalizedHands,
} from "../testing-types";

export default defineScenario({
  meta: {
    id: "initial-state",
    description: "Opening hands are dealt to the correct players",
  },
  given: {
    base: "initial-turn",
  },
  when: {
    steps: [],
  },
  then: {
    assert: ({ gameState, expect }: AssertContext) => {
      const hands = getNormalizedHands(gameState);
      expect(hands["player-1"]?.hand).toEqual([
        "seat-1-flower-1",
        "seat-1-flower-2",
        "seat-1-flower-3",
        "seat-1-skull",
      ]);
    },
  },
});
```

## Runtime Assertion Surface

`then.assert` receives `{ gameState, events, expect }`.

- `events.count(type)` returns number of matching events.
- `events.last(type)` returns the latest matching event.
- `events.all(type)` returns all matching events.
- `expect(actual)` supports `toBe`, `toEqual`, `toBeDefined`, `toContain`, and `toBeGreaterThanOrEqual`.

During scenario assertions, event history includes only scenario steps (base-step events are cleared before scenario execution).

`gameState.globalVariables` and `gameState.playerVariables` are JSON-parsed for assertions.

`testing-types` also exports:

- `getNormalizedHands(gameState)` to read each player hand with sorted card IDs.
- `getNormalizedDecks(gameState)` to read deck contents with sorted card IDs.

## Failure And Staleness Signals

Common failures indicate stale generated bases:

- `Missing test/generated/base-states.generated.ts. Run 'dreamboard test generate' first.`
- `Fingerprint mismatch: base setup changed (...). Run 'dreamboard test generate' to refresh deterministic bases.`
- `Fingerprint mismatch: compiled result changed (...). Run 'dreamboard test generate' to refresh deterministic bases.`
- `Fingerprint mismatch: base setup changed (...); compiled result changed (...). Run 'dreamboard test generate' to refresh deterministic bases.`
- Unknown `given.base` values.

## Common edge cases

When testing out-of-turn behavior in any turn-based game, include these checks:

- Wrong actor attempts a valid action: choose a base where the active player is known, send a normal action from a different player, and assert a turn-violation error (for example `NOT_YOUR_TURN`).
- Unauthorized role attempts an action: if your game has role-restricted actions, attempt the action from an actor without permission and assert the appropriate authorization error.
- Correct actor, wrong phase/state: attempt an action that is valid in one phase/state while the game is in another, and assert a deterministic phase/state rejection.
- Consecutive out-of-turn attempts: send multiple invalid actions from different non-active players and verify each is rejected independently.
- No state mutation on rejection: assert no successful action event was emitted, the game phase/state is unchanged, and the active player pointer is unchanged.

Recommended assertion template:

```ts
expect(events.count("ACTION_REJECTED")).toBe(1);
expect(events.count("ACTION_EXECUTED")).toBe(0);
expect(events.last("ACTION_REJECTED")?.errorCode).toBe("NOT_YOUR_TURN");
expect(gameState.currentState).toBe(expectedState);
expect(gameState.currentPlayerIds[0]).toBe(expectedActivePlayerId);
```
